# PostgreSQL 開発用ロール設計（柔軟性重視）

## 概要

開発環境において、権限管理をシンプルかつ柔軟に保つために  
**グループロール（dev_group）に権限を集約する設計**とする。

- 環境構築（DB・スキーマ・所有権）は管理者が実施
- テーブル作成は開発者が行う
- 権限は個別ユーザーではなくグループロールに集約する

---

## 全体の流れ
管理者作業
1. グループロール（dev_group）作成
2. DB 作成  
3. スキーマ作成
4. 権限付与
5. 開発ログインユーザ作成（dev_group 継承）  

開発者作業
1. SET ROLE
2. テーブル作成

---

### PostgreSQL の階層イメージ
```
サーバー
 └─ データベース（DB）
      └─ スキーマ
           └─ テーブル
```

---

## 管理者作業
### 1. ロール
グループロール（ログイン不可）を作成する。   
```
CREATE ROLE dev_group NOLOGIN;
``` 

### 2. DB 作成
dev_group を所有者としてデータベースを作成する。
- 所有者（OWNER）を指定  
- 文字コード（UTF-8）を指定
```
CREATE DATABASE t4_codegen_dev
    WITH OWNER = dev_group
    ENCODING 'UTF8';
``` 

#### 補足（ロケール）
ロケール指定は環境依存のため注意： 
- LC_COLLATE / LC_CTYPE は OS に依存
- Docker 環境では利用可能なロケールが限定される場合あり、必要に応じて指定する
```
-- 例（環境により失敗する可能性あり）
    LC_COLLATE='ja_JP.UTF-8'
    LC_CTYPE='ja_JP.UTF-8';     
``` 

### 3. スキーマ作成
スキーマの所有者を dev_group にする。  
*⚠️ 注意*   
- *このSQLは「データベース（t4_codegen_dev） に接続した状態」で実行する*

``` 
CREATE SCHEMA IF NOT EXISTS t4_codegen_schema AUTHORIZATION dev_group;
``` 

### 4. 権限付与
※ dev_group は OWNER のため全権限を持ち、本来不要だが、設計の意図を明示するために GRANT を記述している。  

#### DB接続権限
データベースへのアクセス権限を付与
``` 
GRANT CONNECT ON DATABASE t4_codegen_dev TO dev_group;
``` 

#### スキーマ権限
スキーマへのアクセス権限（USAGE）と、テーブル作成権限（CREATE）を付与  
``` 
GRANT USAGE, CREATE ON SCHEMA t4_codegen_schema TO dev_group;
``` 

#### 既存テーブルへの権限
スキーマ内のすべてのテーブルに対してフル権限を付与  
``` 
GRANT ALL PRIVILEGES ON ALL TABLES IN SCHEMA t4_codegen_schema TO dev_group;
``` 

*⚠️ 注意*
- *上記は「実行時点で存在するテーブル」にのみ適用される*
- *新規作成テーブルには自動適用されない*

##### 補足
ALL PRIVILEGES には以下のフル権限が含まれる：
- SELECT（読み取り）
- INSERT（追加）
- UPDATE（更新）
- DELETE（削除）
- TRUNCATE
- REFERENCES
- TRIGGER


### 5. ログインユーザ作成
 ログインユーザーを作成、権限はグループロールを継承
 ```
CREATE USER koko_dev WITH PASSWORD '***';
GRANT dev_group TO koko_dev;
```

## 開発者作業
### 1. SET ROLE
*⚠️ 重要*  
- *SET ROLE を使用する*

開発者は必ず以下を実行してから作業する：
```
SET ROLE dev_group;
```
作業後：
```
RESET ROLE;
```

理由
- テーブルの所有者（OWNER）を dev_group に統一するため
- 個人ユーザー所有になると、他メンバーが操作できなくなるため



### 2. テーブル作成
念のため、現在の実行ロール（SET ROLE の結果）を確認する。
```
SELECT current_user;
```
#### サンプルテーブル
- IDENTITY（自動採番）
- VARCHAR / NUMERIC / BOOLEAN / TIMESTAMP WITH TIME ZONE
- NULLあり/なし
- 複合キーあり
- 外部キーあり

```
CREATE TABLE t4_codegen_schema.mst_customer (
    customer_id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    customer_code VARCHAR(20) NOT NULL UNIQUE,
    customer_name VARCHAR(100) NOT NULL,
    email VARCHAR(255),
    is_active BOOLEAN NOT NULL DEFAULT TRUE,
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP WITH TIME ZONE
);

CREATE TABLE t4_codegen_schema.mst_product (
    product_id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    product_code VARCHAR(20) NOT NULL UNIQUE,
    product_name VARCHAR(100) NOT NULL,
    price NUMERIC(10,2) NOT NULL,
    stock_quantity INTEGER NOT NULL,
    is_deleted BOOLEAN NOT NULL DEFAULT FALSE,
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE t4_codegen_schema.trx_order (
    order_id INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    customer_id INTEGER NOT NULL,
    order_date TIMESTAMP WITH TIME ZONE NOT NULL,
    total_amount NUMERIC(10,2) NOT NULL,
    status VARCHAR(20) NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (customer_id)
        REFERENCES t4_codegen_schema.mst_customer(customer_id)
);

CREATE TABLE t4_codegen_schema.trx_order_detail (
    order_id INTEGER NOT NULL,
    line_no INTEGER NOT NULL,
    product_id INTEGER NOT NULL,
    quantity INTEGER NOT NULL,
    unit_price NUMERIC(10,2) NOT NULL,
    PRIMARY KEY (order_id, line_no),
    FOREIGN KEY (order_id)
        REFERENCES t4_codegen_schema.trx_order(order_id),
    FOREIGN KEY (product_id)
        REFERENCES t4_codegen_schema.mst_product(product_id)
);

```


## 設計のポイントまとめ
- 権限はグループロールに集約する
- オブジェクト所有者は dev_group に統一する
- 開発者は SET ROLE を使用する
- 権限付与は「既存オブジェクトのみ」に注意

## 補足（運用上の注意）
- SET ROLE を忘れると、テーブルの所有者が個人ユーザーになる
- 権限設計を統一するため、作業前後の操作を習慣化する
- チーム開発では所有者統一が非常に重要
